REST API

REST usa HTTP como transporte. Cada recurso tem uma URL e o client usa verbos (GET, POST, PUT, DELETE) para operar sobre eles. A comunicação é stateless — cada request é independente.

1. Client monta request
2. GET /users via HTTP
3. Server processa
4. Response JSON 200 OK
5. Client recebe dados

Server (porta 3001)

Como funciona o REST

  • Cada recurso é identificado por uma URL (ex: /users, /users/1)
  • Verbos HTTP definem a ação: GET (ler), POST (criar), PUT (atualizar), DELETE (remover)
  • Stateless: o server não guarda estado entre requests
  • Resposta tipicamente em JSON com status code HTTP (200, 404, 500...)
  • Simples de cachear via headers HTTP padrão

GraphQL

GraphQL usa um único endpoint e o client envia uma query declarativa pedindo exatamente os campos que precisa. Evita over-fetching e under-fetching de dados.

1. Client escreve query
2. POST /graphql
3. Server valida schema
4. Resolver executa
5. Retorna só o pedido

▼ Query
▲ Data

Server (porta 3002)

Como funciona o GraphQL

  • Único endpoint (geralmente POST /graphql)
  • Client define a estrutura da resposta via query language
  • Schema tipado no server define o que é possível consultar
  • Resolvers mapeiam campos da query para lógica de dados
  • Suporta queries, mutations (escrita) e subscriptions (realtime)

gRPC

gRPC usa Protocol Buffers (binário) sobre HTTP/2. O contrato é definido em um arquivo .proto e o client chama funções remotas como se fossem locais.

1. Contrato .proto definido
2. Client chama RPC
3. Serializa em binário
4. HTTP/2 multiplexado
5. Server deserializa
6. Resposta binária

Server (porta 3003)

Como funciona o gRPC

  • Arquivo .proto define serviços e mensagens (tipagem forte)
  • Código gerado automaticamente para client e server (stubs)
  • Serialização binária (Protocol Buffers) — menor e mais rápido que JSON
  • Usa HTTP/2: multiplexing, streaming bidirecional, compressão de headers
  • Ideal para comunicação entre microserviços com baixa latência
🔌

WebSocket

WebSocket abre uma conexão persistente e bidirecional. Após o handshake HTTP inicial, client e server podem enviar mensagens a qualquer momento sem overhead de novos requests.

1. HTTP Upgrade request
2. Handshake aceito (101)
3. Conexão aberta
4. Client envia msg
5. Server responde
6. Bidirecional contínuo

Client

⇅ Full-duplex
Conexão persistente

Server (porta 3004)

Como funciona o WebSocket

  • Inicia com HTTP Upgrade handshake (status 101 Switching Protocols)
  • Após handshake, conexão TCP fica aberta — sem overhead de reconexão
  • Bidirecional: ambos os lados enviam a qualquer momento
  • Frames leves (2-14 bytes de header vs ~800 bytes HTTP)
  • Ideal para: chat, games, dashboards realtime, notificações push
📨

Message Queue

Na comunicação via fila, o Producer publica mensagens e o Consumer processa de forma assíncrona. Não há conexão direta — o broker intermediário garante entrega e desacoplamento.

1. Producer cria mensagem
2. Publica na fila
3. Broker armazena
4. Consumer consome
5. Processa mensagem
6. Acknowledge (ACK)

Consumer / Broker (porta 3005)

Como funciona a Message Queue

  • Producer e Consumer são desacoplados — não se conhecem diretamente
  • Broker (ex: RabbitMQ, Kafka, SQS) armazena mensagens até serem consumidas
  • Garante entrega mesmo que o consumer esteja offline (durabilidade)
  • Permite processamento assíncrono e em paralelo (scaling horizontal)
  • Consumer envia ACK após processar — fila remove a mensagem
  • Patterns: pub/sub, work queues, request/reply, dead letter
